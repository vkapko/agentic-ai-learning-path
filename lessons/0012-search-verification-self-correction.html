Lesson 0012

When One Pass Is Not Enough

Tangible win: for a hard task, choose between exploring action paths (LATS), generating a reasoning strategy first (Self-Discover), and an independent verifier — and judge what actually makes a second pass add reliability instead of just cost.

The One Idea

Lesson 0005 chose how an agent reasons in one forward pass, and lesson 0007 assumed a single plan was good enough to execute. This lesson covers what to do when one pass is not enough. The local planning notes describe three moves — explore paths before committing, compose a strategy before solving, and verify output before acting — and one rule that governs all of them: an extra pass only helps if it brings something the first pass did not have.

Keep the compact version open: Search, Verification, and Self-Correction Reference.

Minimum Rule

Add an extra reasoning pass only when it buys reliability you cannot get more cheaply, and make every pass bring something new — a different path, a different strategy, or independent evidence — never just a re-read of the same context by the same mind.

Three Moves Past One Pass

Explore

LATS — search paths

A sequence explorer: an internal tree search proposes several next actions, scores them by feedback or rollout, and commits the highest-value one. Use when early mistakes are hard to recover from. Cost: many model calls per real-world action.

Generate

Self-Discover — compose method

A strategy generator: the model first composes a task-specific reasoning structure, then executes linearly by following it. Use for novel problems with no standard solution shape. Risk: less predictable, harder to debug both code and invented logic.

Verify

Second-pass — add verifier

An independent verifier audits planner output against criteria and returns pass/fail with explanation before the action runs. Supports graded verdicts (block vs. notify a human). Risk: false negatives if the verifier shares the planner's flaws.

The Self-Correction Trap

The seductive error is treating any second pass as if it adds reliability. A generator that reviews its own output in the same session reuses the same context and the same blind spots — it tends to rationalize its answer, not catch its mistake. Reflection and Reflexion (lesson 0005) improve results when each loop feeds in new external evidence: a failed test, a tool error, a real observation. Without new evidence, extra passes mostly add cost and false confidence. Independence is the active ingredient — a separate instance, a different model, a rule system, a formal checker, or a human — which is exactly why the Claude foundations guide rates an independent review instance above in-session self-review. Verification is not free reliability; it is reliability if the reviewer can see what the generator could not.

Design Drill: Make The Verifier Independent

A travel agent drafts a flight-plus-hotel itinerary and is about to place reservations. Design a second-pass verifier that is genuinely independent of the planner, not a self-review.

{
  "checks": ["dates contiguous", "hotel night count matches stay", "total within budget", "seats and rooms still available"],
  "independentEvidence": "live availability + price re-query through the booking API, not the planner's cached draft",
  "verdictLevels": ["fatal", "minor", "pass"],
  "rules": {
    "fatal": "block execution and return the failing check to the planner as its next observation",
    "minor": "notify a human-in-the-loop; do not block",
    "pass": "release the reservation action"
  },
  "independenceNote": "separate prompt + external API as ground truth so it does not share the planner's context or blind spots"
}

Design Drill: Pick The Pass

For each scenario, name the single best move — search paths, compose method, add verifier, or single pass — and one line of why.

A) Single pass — low stakes, reversible, an extra pass only adds cost. B) Search paths (LATS) — early mistakes are unrecoverable, so weigh alternatives before committing. C) Compose method (Self-Discover) — no standard structure exists, so generate one before solving. D) Add verifier — high-stakes, irreversible side effect that must be independently checked (and gated by approval) before it runs.

Design Drill: Defend The Downgrade

A teammate proposes a five-iteration self-reflection loop on a document-summarization task, with the same model re-reading its own summary each round and no external check. In two sentences, argue why this mostly adds cost.

Each iteration re-reads the same source through the same model with no new external evidence, so it inherits the same blind spots and tends to rationalize rather than catch errors — five passes mostly multiply latency and tokens. Reliability would improve only by adding something independent: a checklist of required facts verified against the source, a different reviewing instance, or a rule check that the summary omits no flagged section.

Architecture Check

  • Where does the single forward pass actually break — wrong path, missing strategy, or unchecked output?
  • If early mistakes are unrecoverable, is exploring and scoring paths worth the extra model calls?
  • For a novel task, does composing a reasoning structure first beat a generic single pass?
  • Does every added pass bring a new path, a new strategy, or new evidence — or just re-read the same context?
  • Is the verifier independent (separate instance, model, rule, tool, or human), or is it self-review in disguise?
  • What evidence does the verifier hold that the generator did not?
  • Are verdicts graded so minor issues notify a human while fatal ones block execution?
