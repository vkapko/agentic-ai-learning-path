Lesson 0007

Draw The Tool Orchestration Plan

Tangible win: turn a workflow into a small execution plan that marks dependencies, parallel calls, approval gates, failure paths, and the final synthesis step.

The One Idea

After you design individual tool contracts, design how tools are coordinated. A tool orchestration plan is a controller-owned map of what can run now, what must wait, what needs approval, what happens on failure, and what evidence returns to the model. The local planning notes describe this as explicit dependencies: independent tasks can run concurrently, while dependent tasks remain sequential [local planning notes].

Keep the compact version open: Tool Orchestration Plans Reference.

Minimum Rule

Parallelize only when calls do not depend on each other's outputs and do not compete for the same scarce, mutable, or permissioned state. Gate actions when they are irreversible, unsafe, expensive, externally visible, or low confidence.

Five Marks

Inputs

Known now

Record user intent, task state, available identifiers, permissions, and missing facts.

Edges

Wait for

Draw dependency edges where one call needs another call's output before it can run.

Parallel

Run together

Group independent read calls, retrieval calls, or checks that can execute at the same time.

Gates

Stop or approve

Add policy, verifier, confidence, budget, and human approval gates before sensitive actions.

Synthesis

Combine evidence

Define the final result shape and which observations, errors, and approvals are cited.

Two Useful Patterns

A general concurrent execution optimizer compiles a plan into a dependency graph or task DAG, then dispatches independent function calls in parallel. The LLMCompiler paper is the primary source for this style of parallel function calling [LLMCompiler paper].

ReWOO is narrower: the model plans retrieval needs before observations arrive, then the controller fills evidence placeholders and synthesizes a final answer. The local notes frame it as useful for information-heavy tasks where some retrieval calls can be planned up front [ReWOO notes], and the original paper reports token-efficiency gains for that decoupled pattern [ReWOO paper].

Choose The Execution Shape

Pick the smallest execution shape that fits the dependency and risk.

Design Drill: Refund Workflow

A support assistant handles a refund request. It can read order history, read shipment status, read refund policy, check fraud risk, draft a recommendation, create an escalation ticket, and issue refunds under $100 only after eligibility passes.

Known inputs: customer ID, order ID, requested amount, reason. Parallel group: read order history, shipment status, refund policy, and fraud risk because they are independent reads. Then: synthesize eligibility and amount recommendation. Gate: block refund if ineligible, over $100, high fraud risk, identity mismatch, or missing approval. Failure path: retry transient read failures once, then create an escalation ticket with partial evidence. Synthesis: return recommendation, evidence, blocked reasons, approval state, and audit ID.

Design Drill: Dependency Check

For each pair, mark whether it can run in parallel or must run serially.

1. Parallel if both inputs are already known. 2. Serial because the second call needs the resolved customer ID. 3. Parallel if both are read-only and independent. 4. Serial with an approval or policy gate before the refund action.

Architecture Check

  • Which tool calls are independent reads?
  • Which calls require IDs, permissions, or facts produced by earlier calls?
  • Which steps mutate external state or create visible side effects?
  • Which gates are enforced by the controller rather than the prompt?
  • What happens if one parallel call fails but the others succeed?
  • What final evidence bundle is passed back to the model for synthesis?
