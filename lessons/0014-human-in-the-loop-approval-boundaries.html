Lesson 0014

Human-in-the-Loop and Approval Boundaries

Tangible win: for a list of agent actions, place a human gate only where it earns its cost, pick the right kind of interrupt (approve-before, review-after, or escalate-on-trigger), write explicit escalation criteria, and say what a human's feedback actually changes — memory, policy, or weights.

The One Idea

The mission names approval and control boundaries as first-class design surfaces, alongside perception, reasoning, action, memory, and monitoring. A human gate is a control boundary: a point where the agent stops and hands a decision to a person. The local notes frame it plainly — human-in-the-loop inserts human judgment, approval, intervention, or teaching into the loop. The design question is never "should there be a human?" but where, what kind of interrupt, and what the human's feedback changes.

Keep the compact version open: Human-in-the-Loop and Approval Boundaries Reference.

Minimum Rule

Gate an action when the cost of a wrong autonomous action exceeds the cost of waiting for a human — and nowhere else. Each gate names an explicit trigger, an interrupt mode, and an owner who is accountable for the override. Gating everything dies of latency; gating nothing is unsafe.

Where to Gate: The Cost Trade

A gate buys safety and pays in latency and staffing. So place it only where the trade is worth it. The notes give explicit conditions for a human stop: low confidence, safety risk, missing permission, irreversible action, policy conflict, or a novel environment state. Read that list as a cost test — each item is a case where a wrong move is expensive, unrecoverable, or out of the agent's authority. Everything else the agent should just do, because latency and staffing cost are the standing price of every gate.

Three Interrupt Modes

Once you know an action needs a human, choose how the human enters. The mode follows from reversibility and volume, not from how important the action feels.

Approve-before

Block, then act

Agent pauses and waits for a human to approve before the action runs. For irreversible, unsafe, or expensive actions where a wrong move can't be undone. Pays full latency every time.

Review-after

Act, then correct

Agent acts immediately; a human reviews later and corrections feed back. For reversible, high-volume work where latency matters and mistakes are cheap to fix.

Escalate-on-trigger

Run until a line is crossed

Agent proceeds autonomously until a defined trigger fires — low confidence, policy conflict, novel state — then hands off. The default for everything in between.

A fourth flavor is teaching: a human demonstrates or corrects, and that example is captured for next time. It connects directly to monitoring and recovery (lesson 0008) and to escalation by provenance (lesson 0010) — a gate is the deliberate, planned cousin of an emergency stop.

What the Human's Feedback Actually Changes

This is where designs quietly go wrong. The notes are explicit: a human demonstration does not make the model permanently learn a new skill by itself. Instead the system might record a procedural memory, create a skill template, change a threshold or prompt, update a routing rule, or — separately and rarely — trigger fine-tuning. This is exactly the three-updates frame from lesson 0013, now driven by a human:

  • Memory write — "this client always wants the morning slot." A record, scoped to whoever retrieves it.
  • Policy / threshold update — "from now on, refunds over $500 need approval." A rule that binds every run.
  • Model-weight update — RLHF or fine-tuning. Slow, expensive, and not what one correction does.

Two failure modes follow. Feedback leaking into prompts without validation turns a stray correction into durable behavior — the same poisoned-memory risk as lesson 0013. And a missing audit trail means no one can answer who approved what, which matters most when humans override the agent and ownership is already unclear.

Classify the Gate

For each situation, pick the interrupt mode or the system the feedback should change.

Recall Check

From memory: name the six standard escalation triggers that should route a decision to a human, and the three interrupt modes.

Triggers: low confidence, safety risk, missing permission, irreversible action, policy conflict, novel environment state. Interrupt modes: approve-before (block then act), review-after (act then correct), escalate-on-trigger (run until a line is crossed). Teaching/demonstration is a fourth flavor that produces a record, not a weight change.

Design Drill: Place the Gates

A travel-booking agent can take these actions in one run: (a) search flights, (b) hold a refundable seat, (c) charge the client's card for a non-refundable fare, (d) email the client an itinerary, (e) cancel an existing booking, (f) book travel to a region it has never handled before. For each, decide no-gate / approve-before / review-after / escalate-on-trigger, and justify it by the cost of a wrong move.

(a) search flights     -> no gate     (reversible, no side effect)
(b) hold refundable    -> no gate     (cheap to release; reversible)
(c) charge card (NR)   -> approve-before  (expensive + irreversible)
(d) email itinerary    -> review-after    (reversible-ish; latency matters)
(e) cancel booking     -> approve-before  (irreversible loss for the client)
(f) novel region       -> escalate-on-trigger (novel state = low confidence)

The principle is constant: gate where wrong-action cost > wait cost, and let the interrupt mode follow from reversibility and volume.

Design Drill: Write the Escalation Criteria

For the same agent's autonomous zone (the actions you did not hard-gate), write the explicit triggers that should still pause it mid-run and route to a human. Make them checkable, not vague.

Pause and escalate when: model/tool confidence falls below a set threshold; total spend on a booking exceeds a dollar cap; the action would touch a region or fare class not seen before; two memory or tool results conflict and can't be resolved by authority; the caller lacks permission for the requested action; or any action flagged irreversible reaches its execution step. Each is a boolean the runtime can evaluate — not "if it seems risky."

Design Drill: What Does the Feedback Change?

After a run, a human gives three corrections: (1) "This client always prefers aisle seats." (2) "Stop auto-booking non-refundable fares without approval — make that the rule for everyone." (3) "Your seat-selection step keeps mis-parsing this airline's seat map." For each, name whether it should change memory, policy, or weights — and what audit trail it needs.

(1) Memory write — episodic, scoped to that client, with provenance and confidence; audit = who added it, when. (2) Policy/threshold update — a rule binding every run, reviewed and rolled out, with a change record of who approved it. (3) Likely a separate component fix (a parser/perception model) or a procedural-memory note to retry/route differently — and only a candidate for retraining if the pattern persists; never an inline weight change. The shared rule: validate before any correction reaches a prompt, and log who changed what so an override is never anonymous.

Architecture Check

  • Does each gate exist because wrong-action cost beats wait cost — or did it creep in "to be safe" and now taxes every run?
  • Is the interrupt mode (approve-before / review-after / escalate-on-trigger) matched to reversibility and volume, not to how scary the action feels?
  • Are escalation triggers explicit and checkable (confidence, spend cap, permission, irreversibility, conflict, novel state) rather than "if it seems risky"?
  • When a human gives feedback, is it routed to the right system — memory, policy, or a training process — instead of all landing in the prompt?
  • Is feedback validated before it can shape future behavior, so one stray correction can't poison the agent?
  • Is there an audit trail naming who approved or overrode each gated action, and who owns the outcome?
  • What is the fallback when the human is slow or absent — does the agent block, time out, or take the safe default?
