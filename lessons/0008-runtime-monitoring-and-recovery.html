Lesson 0008

Design The Runtime Safety Net

Tangible win: define what an agent run must log, what signals trigger intervention, and how the system recovers from partial failure without hiding uncertainty.

The One Idea

An agent architecture is incomplete until it says how execution is observed and recovered. The local architecture notes warn that agentic systems still need state management, permissions, logging, testing, rollout, recovery, and monitoring, and that every important action needs a feedback path [local architecture notes] [design checklist].

Keep the compact version open: Runtime Monitoring and Recovery Reference.

Minimum Rule

Monitor the run at the same boundary where the controller grants authority: model decision, tool call, policy check, state update, human approval, final answer, and recovery path. If a failure changes coverage or confidence, surface it in the final result.

Five Signals

Trace

What happened

Record task ID, model calls, tool calls, inputs after validation, outputs, approvals, retries, and final state.

Outcome

Did it work

Track success criteria, user-visible completion, blocked reasons, partial results, and unresolved assumptions.

Risk

Should it stop

Watch for unsafe actions, policy conflicts, low confidence, stale state, sensitive data, and approval thresholds.

Resource

Cost and time

Measure latency, token use, tool cost, retry count, queue time, and budget exhaustion.

Recovery

What next

Choose retry, replan, fallback, human escalation, partial answer, rollback, or stop with an explicit explanation.

Failure Is Data

A valid empty result is different from a failed lookup. The Claude architecture reference emphasizes structured error context: failure type, attempted query, partial results, and possible alternatives, so a coordinator can retry, change approach, proceed with coverage gaps, or escalate [Claude error propagation notes].

The local solution-pattern notes make the broader reliability point: production reliability depends on observability, retry semantics, memory consistency, authorization, idempotency, and failure isolation [local solution notes].

Choose The Recovery Move

Pick the smallest recovery action that preserves correctness and control.

Recall Check

From memory: name the five runtime signals to monitor.

Trace, outcome, risk, resource use, and recovery path.

Design Drill: Support Agent Run

A support agent can read customer records, inspect orders, check policy, draft responses, escalate tickets, and process refunds under $100. Design the runtime safety net for one refund request.

Trace fields: request ID, customer ID, order ID, validated tool inputs, policy result, refund amount, approvals, tool errors, retry count, and final status. Stop conditions: identity mismatch, missing order, policy denial, amount over $100, high fraud risk, or repeated tool failure. Retry rules: retry transient read failures once, never retry validation or permission failures as if they were transient. Escalation triggers: customer asks for human help, policy gap, high-risk refund, unavailable required evidence, or no meaningful progress. Final disclosure: cite evidence used, say what was unavailable, and state whether the refund was processed, blocked, or escalated.

Design Drill: One Log Line

Write a compact structured log event for a failed order lookup. Include enough information for recovery, but do not include unnecessary raw customer data.

{
  "event": "tool_error",
  "taskId": "refund-1842",
  "tool": "lookup_order",
  "errorCategory": "transient",
  "isRetryable": true,
  "attemptedInput": { "orderId": "ord_123" },
  "partialResult": null,
  "recoveryHint": "retry once, then escalate with missing order-status evidence"
}

Architecture Check

  • What does success mean for this run, and who can verify it?
  • Which events are logged before and after every tool call?
  • Which failures are transient, validation, permission, policy, or business-rule failures?
  • Which recovery moves are automatic, and which require human review?
  • How does the final response disclose missing evidence or partial completion?
  • Which repeated failures should become design work rather than more retries?
