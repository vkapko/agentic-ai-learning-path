Lesson 0013

Memory Is Data Infrastructure

Tangible win: after an agent finishes a task, decide what to write to long-term memory, classify each record, attach the metadata that makes it governable, and write a retrieval rule that ranks by authority and recency — not similarity alone.

The One Idea

Earlier lessons kept touching memory without designing it: lesson 0005 listed episode memory as a reasoning option, lesson 0010 attached provenance to facts inside one run, and lesson 0011 gave agents a shared-state contract. This lesson designs the store those all depend on. The local notes are blunt about the frame: treat memory as data infrastructure — provenance, permissions, expiry, conflict handling, and observability. Memory is not the chat transcript and it is not learning; it is a governed data store the agent reads from and writes to on purpose.

Keep the compact version open: Memory as Data Infrastructure Reference.

Minimum Rule

Every long-term memory record carries provenance, owner, timestamp, confidence, and expiry — and retrieval ranks records by authority and recency, not by similarity alone. A store without those is a transcript, not memory.

Two Kinds of Durable Memory

The planning notes split durable memory from the short-lived scratchpad that holds the current task's state. The scratchpad is thrown away when the task ends; durable memory survives across sessions and comes in two flavors.

Episodic

What happened

Specific past interactions: requests, responses, feedback, user preferences, history. "User A always books the Innovation Suite and requests vegetarian catering." Answers who wanted what, when.

Procedural

What worked

Action sequences and tool outcomes: which toolchain succeeded, parameters, latency, success rate. "Venue search needs 3 retries; internal booking succeeds first try." Answers how to do it well next time.

Scratchpad

Right now (not durable)

Current task goals, intermediate results, observations. Lives for one run, then discarded. Promoting scratchpad to durable memory without filtering is how stores fill with noise.

The Three-Updates Trap

The costly confusion is treating every "the agent got better" the same way. Three different things can change, and they live in three different systems:

  • Memory update — write a record to the store. Cheap, instant, reversible, scoped to whoever retrieves it. "Remember this user prefers vegetarian."
  • Policy / threshold update — change a rule, prompt, escalation trigger, or evaluator. Affects every run, needs review and rollout. "Always require approval above $500."
  • Model-weight update — fine-tune or retrain. Slow, expensive, hard to reverse, changes the model itself. Not what happens when an agent "learns from feedback" via memory.

Reflexion and in-context learning improve behavior by writing memory and re-injecting it into the prompt — not by touching weights. Conflating the three leads to real mistakes: putting a security rule in episodic memory where one user's record can override it, or reaching for fine-tuning when a memory write would do. Match the change to the system that should own it.

Retrieval Is the Hard Part

Writing is easy; retrieving the right record under a token budget is where memory fails. A pure similarity search "optimizes similarity but ignores authority or recency" — it will happily surface a stale, low-confidence, or out-of-scope memory because it reads like the query. Good retrieval scores on more than one axis. The Generative Agents memory stream is the canonical example: it ranks by recency, importance, and relevance together. Add authority (a policy record outranks a user anecdote) and permission (never retrieve a memory the current caller is not allowed to see), and only then inject the top few into the prompt — because, as lesson 0010 noted, a long context window does not read everything uniformly.

Classify the Record

For each item an agent might store, name what kind of record it is and which system should own the change.

Recall Check

From memory: name the five pieces of metadata every durable record should carry, and the two ranking axes retrieval must add beyond similarity.

Metadata: provenance (source), owner, timestamp, confidence, expiry. Retrieval must add at least authority (how trusted/binding the record is) and recency (how current it is) on top of relevance — and filter by permission before anything is injected.

Design Drill: Design the Write-Back

A scheduling agent just booked a team offsite for a returning client. The run produced: the client's request, the venue it chose, a note that the catering tool failed twice then succeeded, the client's emailed thanks, and the human approver's sign-off on the spend. Decide what to write to durable memory, what to drop, and the type plus metadata for each kept record.

[
  { "record": "client prefers this venue + offsite format",
    "type": "episodic", "keep": true,
    "meta": { "owner": "clientId", "source": "this run", "confidence": "med",
              "timestamp": "2026-06-20", "expiry": "review in 12mo" } },
  { "record": "catering tool: 2 failures then success; budget retries",
    "type": "procedural", "keep": true,
    "meta": { "owner": "ops", "source": "tool logs", "confidence": "high",
              "timestamp": "2026-06-20", "expiry": "supersede on tool change" } },
  { "record": "approver signed off on the spend",
    "type": "episodic/approval", "keep": true,
    "meta": { "owner": "audit", "source": "human approval", "confidence": "high",
              "expiry": "retain per policy" } },
  { "record": "client's thank-you email",
    "keep": false, "why": "no future decision depends on it; scratchpad only" }
]

Note: the spend rule itself ("offsites need approval") is a policy, not a memory write — it should already live as a rule, not be re-learned per run.

Design Drill: Write the Retrieval Rule

The same agent starts a new task for a different client. Its memory store holds thousands of records. Write the rule that decides which memories get injected into the prompt — and make it resistant to surfacing the wrong account's data or a stale preference.

Hard filters first: drop any record the current caller lacks permission to see, any past its expiry, and any not scoped to this client (no cross-account leakage). Then rank the survivors on relevance × recency × authority (policy and approval records outrank user anecdotes), with a small confidence weight. Inject only the top 3–5 to respect the token budget and avoid burying the binding facts. On conflict, prefer higher authority, then more recent; if a high-stakes conflict remains unresolved, escalate rather than guess.

Design Drill: Defend Against Poisoned Memory

A teammate wants the agent to auto-write every user message verbatim into long-term episodic memory "so it never forgets context." In two sentences, name the failure this invites and the cheapest guardrail.

Verbatim auto-write lets a bad, malicious, or injected message become a durable record that later retrieval treats as truth — bad memories poison future behavior, and stored text becomes a prompt-injection vector. The cheapest guardrail is to write only filtered, summarized records with provenance and confidence, low-confidence/unverified entries expiring fast, and authority ranking so a stray user anecdote can never outrank a policy record.

Architecture Check

  • Is this a memory write, a policy change, or a weight update — and is it going to the system that should own it?
  • Does every durable record carry provenance, owner, timestamp, confidence, and expiry?
  • Is the scratchpad kept separate from durable memory, with a deliberate filter between them?
  • Does retrieval rank on authority and recency, not similarity alone — and filter by permission first?
  • What happens when two records conflict? Who wins, and when does the agent escalate instead?
  • Can a stored user message act as a prompt-injection vector when it is retrieved later?
  • Is there an expiry and retention rule, so the store does not grow stale or violate privacy?
