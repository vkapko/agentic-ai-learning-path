Lesson 0011

Divide The Work, Keep One Truth

Tangible win: pick a coordination shape for a multi-agent task, assign worker roles and tool boundaries, and design the shared-state contract that keeps every agent reading the same truth.

The One Idea

Lesson 0004 answered whether to add agents. This lesson answers how they coordinate. A coordination pattern is three decisions: who decomposes the goal, who does the work, and where the single source of truth lives. The local patterns notes describe three coordination shapes — orchestrator-worker, specialized expert team, and hierarchical systems — and one cross-cutting requirement: coherent state and collective memory.

Keep the compact version open: Multi-Agent Coordination Reference.

Minimum Rule

Pick the lightest coordination shape that matches the task's dependency structure, give every worker a narrow role and a narrow tool set, and give the whole system one consistent place to read and write shared state with provenance, ownership, and conflict rules.

Three Coordination Shapes

Fan out

Orchestrator-worker

One orchestrator decomposes work and delegates units to workers, then merges results. Best for parallelizable subtasks and narrow-tool workers needing centralized tracking. Risks: orchestrator bottleneck, inconsistent worker results, distributed retries.

Deliberate

Specialized expert team

Several domain agents collaborate on one decision and expose a single interface. Best when a subtask needs multiple forms of domain reasoning. Needs protocols for disagreement, source ranking, and final authority. Risks: cost, latency, groupthink.

Nest

Hierarchical

Top orchestrators, sub-orchestrators, workers, and teams manage very large or naturally nested workflows. Scales coordination but is not automatically more reliable. Risks: communication overhead, state drift between layers, slow failure diagnosis.

The Shared-State Trap

The hard part of multi-agent systems is rarely the decomposition; it is keeping a coherent view of task state, observations, decisions, and shared knowledge. A central shared memory can be the source of truth, but it can also become a bottleneck or single point of failure [coherent state notes]. Treat memory as data infrastructure, not a chat log: every record carries provenance, timestamp, owner, confidence, and expiry, and the store enforces access control and conflict resolution. Beware retrieval that optimizes similarity but ignores authority or recency, and beware prompt injection through stored content.

Choose The Coordination Move

Pick the move that matches the task's dependency structure and keeps shared state trustworthy.

Design Drill: Orchestrator-Worker Decomposition

A research assistant must produce a competitor brief: gather pricing, gather feature lists, gather recent news, then synthesize. Design the orchestrator-worker decomposition.

{
  "orchestratorOwns": ["goal", "task plan", "progress", "final synthesis", "escalation"],
  "workers": [
    { "role": "pricing", "tools": ["pricing_api"], "returns": "structured price records" },
    { "role": "features", "tools": ["web_fetch"], "returns": "feature list with sources" },
    { "role": "news", "tools": ["news_search"], "returns": "dated headlines with links" }
  ],
  "parallel": ["pricing", "features", "news"],
  "mergeProtocol": "validate each worker result schema, attach source IDs, reject unparseable returns",
  "failureHandling": "retry a failed worker once; on second failure mark coverage gap and proceed partial"
}

Design Drill: Collective Memory Record

The news worker found a price-cut headline that the pricing worker's API has not yet reflected. Write the shared-memory record so a later agent can resolve the conflict.

{
  "key": "competitor.acme.flagship_price",
  "value": "39.00 USD (announced cut, not yet in API)",
  "owner": "news_worker",
  "provenance": "news_search:article_5521",
  "timestamp": "2026-06-19T14:02:00Z",
  "confidence": 0.6,
  "expiry": "2026-06-26",
  "conflictNote": "pricing_api still reports 49.00 USD as of 2026-06-18; reconcile before publishing"
}

Design Drill: Defend The Downgrade

A teammate proposes a three-layer hierarchy for a task with four independent lookups and one synthesis step. In two sentences, argue for a simpler shape.

The dependency structure is a flat fan-out — four independent lookups feeding one merge — so a single orchestrator with four workers covers it. A hierarchy would add sub-orchestrator layers that bring communication overhead, state drift between layers, and slower failure diagnosis without any extra parallelism or domain reasoning to justify them.

Architecture Check

  • Is the task a flat fan-out, a multi-domain decision, or a deeply nested workflow?
  • What is the narrowest role and tool set each worker needs?
  • Which subtasks can run in parallel, and which have real dependencies?
  • How are worker results validated and merged, and what happens on partial failure?
  • Where is the single source of truth, and is it a bottleneck or single point of failure?
  • Do shared-memory records carry provenance, owner, confidence, and expiry?
  • What rule resolves conflicting writes — and does it weigh authority, not just recency?
