Lesson 0003

Implement the Loop in Python

Tangible win: choose when to use the Claude Python SDK directly and when to use LangGraph, then identify where the agentic control loop lives in each example.

The One Idea

The Claude Python SDK is the direct client for Anthropic's Messages API. The official docs show sync and async clients, streaming, retries, error handling, token counting, and tool helpers for Python 3.9+ [Anthropic Python SDK]. Use it when your application code owns orchestration: policy checks, tool execution, persistence, approvals, and stop conditions.

LangGraph is different. It is a graph runtime for making agent control flow explicit. The official quickstart builds a calculator agent with a model node, tool node, conditional edge, and return edge back to the model until the graph stops [LangGraph quickstart]. Use it when the loop itself is the architecture you want to inspect, test, checkpoint, and evolve.

Keep the compact version open: Python Agent Examples Reference.

Decision Rule

Claude SDK

Application-owned loop

Best for a narrow service where you already know the workflow and want direct API control. The SDK call is a component; your code supplies state, tool permissions, retries, monitors, and approvals.

LangGraph

Graph-owned loop

Best when you need visible nodes and transitions: model call, tool execution, conditional routing, retry, approval, memory, checkpointing, or subgraphs.

Claude SDK: Direct Call

This is useful, but it is not automatically agentic. It produces an answer and stops unless your application wraps it in a loop.

from anthropic import Anthropic

client = Anthropic()

message = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=300,
    messages=[
        {
            "role": "user",
            "content": "Classify this system as predictive, generative, or agentic: a model drafts a support reply.",
        }
    ],
)

print(message.content[0].text)

Claude SDK: Tool Helper

The SDK also provides a tool runner helper. It can call a Python function and feed the result back to Claude on the next iteration. That is closer to a reason-act-observe loop, but the risky boundaries still belong outside the model.

import json
from anthropic import Anthropic, beta_tool

client = Anthropic()

@beta_tool
def lookup_refund_policy(order_total: int) -> str:
    """Look up refund policy for an order total.
    Args:
        order_total: Order value in whole dollars.
    """
    return json.dumps({"auto_refund_limit": 100, "order_total": order_total})

runner = client.beta.messages.tool_runner(
    model="claude-opus-4-8",
    max_tokens=500,
    tools=[lookup_refund_policy],
    messages=[
        {
            "role": "user",
            "content": "Can this $80 refund be handled automatically?",
        }
    ],
)

for message in runner:
    print(message)

Production design question: where would you enforce the refund cap? Correct answer: outside the model, before any money-moving API call.

Claude SDK: Manual Stop Condition

The tool runner hides the loop. When your application owns orchestration you write the loop yourself, and the stop condition is explicit: keep going while stop_reason == "tool_use", otherwise return the final answer. This is the SDK equivalent of LangGraph's should_continue.

import json
from anthropic import Anthropic

client = Anthropic()

tools = [
    {
        "name": "lookup_refund_policy",
        "description": "Look up refund policy for an order total.",
        "input_schema": {
            "type": "object",
            "properties": {"order_total": {"type": "integer"}},
            "required": ["order_total"],
        },
    }
]

def run_tool(name, args):
    if name == "lookup_refund_policy":
        return json.dumps({"auto_refund_limit": 100, "order_total": args["order_total"]})
    raise ValueError(f"unknown tool: {name}")

messages = [{"role": "user", "content": "Can this $80 refund be handled automatically?"}]

while True:
    message = client.messages.create(
        model="claude-opus-4-8",
        max_tokens=500,
        tools=tools,
        messages=messages,
    )

    # Stop condition: the model is done unless it asked to use a tool.
    if message.stop_reason != "tool_use":
        print(message.content[0].text)
        break

    messages.append({"role": "assistant", "content": message.content})
    tool_results = []
    for block in message.content:
        if block.type == "tool_use":
            result = run_tool(block.name, block.input)  # validate + cap here
            tool_results.append(
                {"type": "tool_result", "tool_use_id": block.id, "content": result}
            )
    messages.append({"role": "user", "content": tool_results})

The while True never spins forever: every turn either resolves tool calls and loops, or hits a non-tool_use stop reason and breaks. Other stop reasons (end_turn, max_tokens, stop_sequence, refusal) all exit the loop here.

LangGraph: Explicit Agent Loop

In LangGraph, the loop is visible in the graph: model node, tool node, conditional edge, and return edge. That makes it easier to reason about state, stop conditions, retries, and observability.

import operator
from typing import Annotated, Literal
from typing_extensions import TypedDict

from langchain.chat_models import init_chat_model
from langchain.messages import AnyMessage, HumanMessage, SystemMessage, ToolMessage
from langchain.tools import tool
from langgraph.graph import END, START, StateGraph

model = init_chat_model("claude-sonnet-4-6", temperature=0)

@tool
def add(a: int, b: int) -> int:
    """Add two integers."""
    return a + b

tools = [add]
tools_by_name = {tool.name: tool for tool in tools}
model_with_tools = model.bind_tools(tools)

class MessagesState(TypedDict):
    messages: Annotated[list[AnyMessage], operator.add]

def llm_call(state: MessagesState):
    response = model_with_tools.invoke(
        [SystemMessage(content="Use tools for arithmetic.")] + state["messages"]
    )
    return {"messages": [response]}

def tool_node(state: MessagesState):
    outputs = []
    for tool_call in state["messages"][-1].tool_calls:
        tool_result = tools_by_name[tool_call["name"]].invoke(tool_call["args"])
        outputs.append(ToolMessage(content=str(tool_result), tool_call_id=tool_call["id"]))
    return {"messages": outputs}

def should_continue(state: MessagesState) -> Literal["tool_node", END]:
    if state["messages"][-1].tool_calls:
        return "tool_node"
    return END

builder = StateGraph(MessagesState)
builder.add_node("llm_call", llm_call)
builder.add_node("tool_node", tool_node)
builder.add_edge(START, "llm_call")
builder.add_conditional_edges("llm_call", should_continue, ["tool_node", END])
builder.add_edge("tool_node", "llm_call")

agent = builder.compile()
result = agent.invoke({"messages": [HumanMessage(content="Add 3 and 4.")]})
print(result["messages"][-1].content)

Recall Check

From memory: in one sentence, explain the difference between using the Claude SDK directly and using LangGraph for an agent.

The Claude SDK gives direct model and tool-call access, while LangGraph makes the control loop explicit as stateful nodes and edges with a visible stop condition.

Design Drill

You are building a refund assistant. It may inspect order history and policy, but issuing money requires a hard cap and audit log. Which implementation surface would you start with, and where would the approval boundary live?

Surface: either can work, but LangGraph is a strong default once inspection, approval, action, observation, and recovery become separate states. Approval boundary: outside the model, before the payment/refund API call. Why: the LLM can recommend or prepare the action, but the controller must enforce caps, log the decision, and stop or escalate.

Architecture Check

  • Does this code show the environment being observed, or only a prompt being answered?
  • Which function can change external state?
  • Where are tool arguments validated before execution?
  • Where is the stop condition expressed?
  • Can you log every model call, tool call, approval, and final state?
