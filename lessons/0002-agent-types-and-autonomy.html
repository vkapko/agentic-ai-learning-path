Lesson 0002

Pick the Simplest Agent That Fits

Tangible win: classify a system as simple reflex, model-based reflex, goal-based, or utility-based, justify the smallest type that fits the workflow, and name the extra guardrails each step up the autonomy ladder demands.

The One Idea

The four classic agent types are not a quality ranking. They are a ladder of mechanism: each rung adds capability and cost. A simple reflex agent maps the current percept to an action. A model-based reflex agent adds internal state so it can act on more than the latest input. A goal-based agent adds search or planning to reach a desired state. A utility-based agent adds a scoring function to compare outcomes when several plans succeed. Design discipline is choosing the lowest rung that still solves the task, because every rung up adds failure modes you must monitor.

Autonomy is the second axis. It is scoped, not binary: an agent may be autonomous only within a narrow domain, tool set, permission boundary, or time window. Risk rises as authority, tool access, and time horizon expand — so broader autonomy must be paid for with stronger guardrails and observability.

The Ladder

Simple reflex

Percept to action

Condition-action rules on the current input. No history, no plan. Brittle when the environment is partially observable; easy to over-label as "AI" when it is rule execution.

Model-based reflex

State-aware rules

Maintains an internal model of relevant state, then applies rules against it. Handles incomplete inputs, but risks stale state and unversioned model assumptions.

Goal-based

Plan toward a state

Searches or plans action sequences that reach a desired state. Fits multi-step tasks, but plans go stale and goals can omit safety, legal, or user constraints.

Utility-based

Compare outcomes

Scores competing outcomes when many plans succeed, balancing cost, speed, reliability, and risk. Powerful, but a bad utility function optimizes the wrong proxy.

Classify by Type

Pick the simplest agent type that the description actually requires. A more capable type is rarely "wrong," but the question is which mechanism the workflow needs.

Recall Check

From memory, in one sentence each: what single capability does each rung add over the one below it?

Simple reflex: baseline — map the current percept to an action. Model-based reflex: adds internal state so rules act on more than the latest percept. Goal-based: adds search or planning toward a desired end state. Utility-based: adds a scoring function to compare and rank successful outcomes.

Design Drill: Justify the Smallest Type

For each workflow, name the smallest agent type that fits and the one fact that forces that rung (and rules out the one below).

Workflows: (a) auto-tag a ticket "urgent" when it contains a keyword; (b) approve an expense only if it fits this month's running budget; (c) assemble a multi-leg travel itinerary that satisfies dates and budget; (d) choose between three valid itineraries by traveler preference.

a: simple reflex — one percept (keyword) maps to one action; no state needed. b: model-based reflex — the decision depends on running budget state, not just the current expense. c: goal-based — it must plan a sequence that reaches a valid itinerary, so a rule on current state is not enough. d: utility-based — several itineraries already satisfy the goal, so the work is scoring them against weighted preferences.

Design Drill 2: Pay for Autonomy

Autonomy is scoped. Take one agent and widen its scope one step at a time; for each step, name the guardrail or monitor that step now requires. Notice that each step expands a different dimension of risk — authority (Step 1), time horizon via an uncapped running total (Step 2), and tool access to a new action (Step 3).

Agent: a model-based reflex restocker that today recommends purchase orders for a human to approve. Step 1: let it place orders under $500 automatically. Step 2: remove the dollar cap. Step 3: let it also switch suppliers on its own.

Step 1 (auto under $500): a hard spend limit enforced outside the model, per-order and rolling-total caps, plus an audit log of every auto-placed order. Step 2 (no cap): rate limits and anomaly detection on order size and frequency, a kill switch, and human review thresholds — removing the cap removes the cheapest containment, so containment must move into monitoring. Step 3 (switch suppliers): an approved-vendor allowlist, contract and compliance checks before any switch, and alerting on supplier concentration — new authority opens new failure modes (fraudulent or non-compliant vendors) that the previous guardrails never covered.

Architecture Check

Before fixing an agent type or expanding its autonomy, ask:

  • What is the highest rung the workflow truly needs — and what would break if you dropped one rung?
  • If you reached for goal-based or utility-based, is there genuinely more than one successful outcome to plan or rank?
  • What is the current autonomy scope: which domain, tools, permissions, and time window?
  • For each expansion of authority, tool access, or time horizon, which new failure mode appears, and what guardrail or monitor contains it?
  • Is any containment relying on a single limit that a later step will remove?
