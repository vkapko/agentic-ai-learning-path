Lesson 0016

Claude Code Configuration and CI/CD Workflows

Tangible win: given a convention, place it at the right Claude Code surface — root CLAUDE.md, a directory CLAUDE.md, a path-scoped rule, a skill, or a command — and at the right sharing level (project vs user); choose plan mode vs direct execution by scope; and wire a CI review so its findings are trustworthy.

The One Idea

This lesson grounds an abstraction the course has used repeatedly. Lesson 0010 asked whether a rule should be enforced programmatically or guided by prompt; lesson 0006 asked how to scope authority to an agent. Claude Code's configuration system is a concrete instance of both. Every config artifact answers two questions: when does it load (always / only on matching paths / only on demand) and who gets it (shared through version control, or personal to one machine). Placing configuration well is the same skill as governing context (lesson 0013): the right instruction reaches the agent exactly when relevant, and nothing else clutters the window.

Keep the compact version open: Claude Code Configuration and CI/CD Reference. This is the applied counterpart to the conceptual patterns — Domain 3 of the Claude Certified Architect reference.

Minimum Rule

Place each instruction at the narrowest scope that still covers everywhere it must apply, and at the sharing level that matches who must follow it. Always-loaded standards go in CLAUDE.md; conventions for files scattered across the tree go in a path-scoped rule; on-demand workflows go in a skill. Project scope is shared via the repo; user scope stays on your machine.

The Configuration Surfaces

Five surfaces, distinguished by when they load and who they reach. The hierarchy is user-level (~/.claude/), project-level (root CLAUDE.md or .claude/), and directory-level (subdirectory CLAUDE.md).[3.1]

Root CLAUDE.md

Always loaded

Universal, always-in-context standards for the whole project. Shared via version control. The cost of every line is permanent context, so keep it to what truly applies everywhere.

Directory CLAUDE.md

Loaded in that subtree

Conventions for one area or package, loaded when working there. Use @import to pull in shared standards files and keep each one modular.[3.1]

.claude/rules/

Loaded on matching paths

A rule file with YAML paths globs (e.g. ["**/*.test.*"]) loads only when editing matching files — even when those files are spread across many directories. The win a per-directory CLAUDE.md can't give.[3.3]

.claude/skills/

Loaded on demand

A task workflow invoked when needed, not always in context. Frontmatter scopes it: context: fork isolates verbose output, allowed-tools restricts authority, argument-hint prompts for inputs.[3.2]

A fifth surface is slash commands in .claude/commands/ — saved prompts, project-scoped (shared) or user-scoped (personal). The deciding axis across all five is the same: CLAUDE.md is always-loaded standards; skills and commands are on-demand; rules are conditional by path.[3.2]

Shared vs. Personal: The Hierarchy Trap

The most common configuration bug is a scope mismatch. An instruction in ~/.claude/CLAUDE.md applies only to you — it is never shared with teammates through the repo. So a standard that "everyone follows" but lives at user level silently fails for every new team member.[3.1] The fix is to ask not "where is it convenient to put this?" but "who is accountable for following it?" Team standard → project scope (committed). Personal preference → user scope (yours alone). The /memory command shows which files are actually loaded, so you can diagnose inconsistent behavior across sessions.[3.1]

Plan Mode vs. Direct Execution

This is the plan-and-execute decision applied to a coding agent. Plan mode explores the codebase and designs an approach before any change — for large-scale, multi-file, or architecturally consequential tasks where there are several valid approaches and rework is costly. Direct execution is right for well-scoped changes with a clear path (a single-file fix with a known stack trace).[3.4] The two combine: plan to investigate, then execute the planned approach. For verbose discovery, the Explore subagent isolates the noise and returns a summary — the same context-hygiene move as context: fork.

Claude Code in CI/CD

A CI pipeline is a non-interactive agentic loop. Three pieces make it work:[3.6]

  • -p / --print — runs Claude Code non-interactively: it processes the prompt, writes to stdout, and exits without waiting for input. Without it, the pipeline hangs.
  • --output-format json with --json-schema — forces machine-parseable structured findings, the same reliability move as structured output via tool use (lesson 0009) — so a script can post them as inline PR comments.
  • CLAUDE.md as the context channel — testing standards, fixture conventions, and review criteria documented there reach the CI-invoked agent and cut low-value output.

The reliability point that ties back to lesson 0012: the same session that generated code is weaker at reviewing its own changes than an independent review instance. A trustworthy CI review runs fresh, not as a continuation of the session under review — independence is what makes the verifier meaningful.[3.6]

Place the Configuration

For each scenario, pick the surface or mode that loads exactly when relevant and reaches exactly the right people.

Root CLAUDE.md — always loaded, project-shared. Directory CLAUDE.md — loaded in that subtree, project-shared. .claude/rules/ with paths globs — loaded only on matching files (can span directories), project-shared. .claude/skills/ — on demand, project or user scope (context: fork, allowed-tools, argument-hint). .claude/commands/ — on demand, project (shared) or user (personal). User-level (~/.claude/) is never shared via the repo.

Design Drill: Place Each Convention

A team has these conventions. For each, choose a surface (root CLAUDE.md / directory CLAUDE.md / path-scoped rule / skill / command) and a scope (project vs user), and justify it by when it must load and who must follow it.

(a) "All commit messages follow Conventional Commits." (b) "Terraform files use our module-naming standard" — terraform is one folder. (c) "Every test file, anywhere, must have a matching snapshot." (d) "A repeatable workflow to scaffold a new microservice." (e) "My own preference to always explain edits before applying them."

(a) commit messages   -> root CLAUDE.md (project)  always applies, everyone
(b) terraform         -> directory CLAUDE.md OR rule paths:["terraform/**"]
                         (one folder -> either works; rule if it may spread)
(c) test files        -> path-scoped rule, paths:["**/*.test.*"] (project)
                         scattered across dirs -> only a glob rule covers it
(d) scaffold service  -> skill in .claude/skills/ (project)  on-demand workflow
(e) explain-first     -> user-scoped (~/.claude/) personal, not in the repo

The constant: narrowest scope that still covers everywhere it applies; project if the team must follow it, user if only you.

Design Drill: Wire a CI Review

Sketch how an automated PR review should run in CI so that it (1) does not hang, (2) returns machine-parseable findings a script can post as inline comments, (3) uses the team's review criteria, and (4) is meaningfully independent. Name the flags and the one architectural choice.

Run with -p (non-interactive, so it exits instead of hanging) and --output-format json --json-schema <schema> so findings are structured for automated posting. Provide review criteria, testing standards, and fixtures through CLAUDE.md to cut false positives. Run the review as a fresh, independent instance — not a continuation of the session that wrote the code — because a generator reviewing itself is the weak verifier from lesson 0012. When re-running after new commits, include prior findings in context and instruct it to report only new or still-unaddressed issues.

Architecture Check

  • Does each instruction sit at the narrowest scope that still covers everywhere it must apply — or is it in always-loaded CLAUDE.md when a path-scoped rule would do?
  • Is anything a "team standard" actually living at user scope, where teammates will never receive it?
  • For conventions on files scattered across the tree, are you using a glob rule rather than copies of a directory CLAUDE.md?
  • Do verbose or destructive skills use context: fork and allowed-tools to keep context clean and authority scoped?
  • Is plan mode reserved for large/architectural/multi-approach work, and direct execution used for clear, well-scoped changes?
  • Does the CI review run with -p and structured output, draw criteria from CLAUDE.md, and run as an independent instance rather than reviewing its own session?
