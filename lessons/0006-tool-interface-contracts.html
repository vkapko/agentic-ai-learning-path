Lesson 0006

Design Tools As Control Boundaries

Tangible win: turn a vague agent capability into a tool contract with purpose, schema, permissions, side effects, error states, and observability.

The One Idea

A tool is not just an API wrapper. In an agentic system, it is a control boundary between model intent and real-world effect. The local architecture notes say to separate agent logic from tool execution, policy enforcement, and human approval, and to treat tools as permissioned interfaces with schemas and side effects [local architecture notes] [design checklist].

Keep the compact version open: Tool Interface Contracts Reference.

Minimum Contract

Every production-facing tool should say what it does, when to use it, when not to use it, what arguments it accepts, what it returns, what it can change, who may call it, how expensive or slow it is, and how failures are reported.

Six Fields

Purpose

Job and boundary

Describe the capability, matching examples, and nearby tools it should not replace.

Schema

Typed input, output

Define required fields, optional fields, enums, nullable fields, and normalized result shape.

Authority

Permission scope

Name who or what may call it, under which task state, and which approvals are required.

Effects

Side effects

Mark read-only, reversible, externally visible, expensive, irreversible, or safety-sensitive actions.

Failures

Structured errors

Return category, retryability, attempted inputs, partial result, and recovery hints.

Telemetry

Observable use

Log caller, arguments after validation, result summary, latency, cost, errors, approvals, and correlation IDs.

Tool Descriptions Drive Selection

The local solution-pattern notes say tool descriptions should include capability, schema, permissions, side effects, cost, latency, and failure modes, and that the controller should validate tool choice and arguments before execution [tool discovery notes]. This matters most when two tools sound similar.

Claude's architecture reference makes the same point with a support example: minimal descriptions like "retrieves customer information" and "retrieves order details" cause selection mistakes, while richer descriptions with input formats, example queries, edge cases, and boundaries improve reliability [Claude architecture reference].

Choose The Contract Fix

Pick the smallest contract change that addresses the described failure.

Recall Check

From memory: list the six parts of a useful tool contract.

Purpose and boundaries; input and output schema; authority and permissions; side effects; structured failure states; telemetry and audit fields.

Design Drill: Refund Tool

You are designing process_refund. It can issue money to a customer, so it is externally visible and financially sensitive. Draft the contract boundary.

Purpose: issue an approved refund for a specific order; not for eligibility checks. Arguments: order ID, customer ID, amount, currency, reason code, approval ID. Allowed when: policy check passed, amount is within role limit, identity is verified, and approval ID exists when required. Blocked when: amount exceeds authority, order is missing, policy conflicts, duplicate refund risk exists, or identity is unverified. Returns: refund ID, status, amount, timestamp, and audit correlation ID. Errors: validation, permission, business-rule, duplicate, transient, and provider failure, each with retryability and recovery hint.

Design Drill: Similar Tools

Write one sentence each that separates these two tools: get_customer and lookup_order.

get_customer: use for account identity, contact details, account status, and customer-level history from a customer ID or verified email. lookup_order: use for order status, delivery events, line items, return eligibility, and payment/refund state from an order ID or customer/order pair.

Architecture Check

  • Is the tool read-only, reversible, externally visible, expensive, irreversible, or safety-sensitive?
  • What makes this tool different from nearby tools?
  • Which arguments are required, optional, nullable, or enum-constrained?
  • Which policy checks happen before execution, outside the model?
  • What error categories can callers distinguish?
  • What result fields should enter model context, and what should stay in logs or storage?
