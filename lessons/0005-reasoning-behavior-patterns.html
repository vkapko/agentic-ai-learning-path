Lesson 0005

Choose the Smallest Reasoning Pattern

Tangible win: select one-shot generation, decomposition, ReAct, reflection, or episode memory based on the pressure in the workflow.

The One Idea

Reasoning patterns are not maturity levels. They are tools for different pressures. Use a more involved pattern only when the task needs decomposition, live observations, quality review, or learning from repeated attempts. The local solution-pattern notes warn that visible reasoning can be plausible but unfaithful, so production systems should log structured decisions, tool calls, observations, and verifier results instead of treating a reasoning trace as proof [local solution notes].

Keep the compact version open: Reasoning Behavior Patterns Reference.

Minimum Rule

Start with the simplest pattern that exposes the needed control boundary. Add ReAct only when observations change the next action. Add reflection only when a separate critique can improve a draft, action, or failure recovery. Add episode memory only when future attempts should use prior feedback.

Five Patterns

One-shot

Single pass

Use when the task is low-risk, bounded, and does not require external state. The controller still owns validation and final delivery.

Decomposition

Think then answer

Use when a task benefits from being broken into steps. Prefer hidden reasoning or concise summaries over raw chain-of-thought.

ReAct

Act then observe

Use when the next decision depends on tool results or environment feedback. The controller mediates tools, permissions, and stop conditions.

Reflection

Critique then revise

Use when a draft, action, or failure needs a review pass. Reflection improves odds; it does not guarantee correctness.

Episode memory

Remember feedback

Use for repeated tasks where prior corrections should influence later attempts. Store provenance, expiry, and confidence with memories.

ReAct Is A Loop

The ReAct paper describes interleaving reasoning and acting so the model can use observations from an environment to guide later steps [ReAct paper]. In architecture terms, the important part is not the visible reasoning text. It is the controlled cycle: decide, call a tool, observe, update state, and either continue or stop.

Claude's architecture notes make the stop condition concrete: continue when stop_reason is "tool_use", execute the requested tool through the controller, send the result back, and stop on final-turn conditions such as "end_turn" [Claude architecture notes].

Choose the Pattern

Pick the smallest pattern that fits the described pressure.

Recall Check

From memory: explain why ReAct is more than "chain-of-thought plus tools."

ReAct is a reason-act-observe loop: each tool result or environment observation can change the next decision. The controller must mediate tool access, validate arguments, handle errors, update state, and enforce the stop condition.

Design Drill: Refund Assistant

A refund assistant drafts a decision for a $240 order. It can read order history, read refund policy, and create an escalation ticket. It cannot issue money directly.

Pattern: ReAct, because order and policy observations change the next step. Controller owns: tool permissions, argument validation, policy cap enforcement, audit logs, ticket creation, and stop conditions. Model may decide: which read-only check to request next and whether the gathered evidence supports escalation. Stop condition: final recommendation, escalation ticket created, tool failure escalated, or max turns reached.

Architecture Check

  • What pressure justifies the reasoning pattern: decomposition, observation, review, or repeated feedback?
  • What should stay hidden, and what concise rationale should be shown?
  • Which tool calls can change the environment?
  • What validates tool names, arguments, permissions, and returned errors?
  • What observations are saved as task state, and what feedback becomes longer-term memory?
  • What stop condition prevents runaway reasoning or tool use?
