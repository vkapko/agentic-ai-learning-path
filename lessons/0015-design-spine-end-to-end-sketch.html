Lesson 0015 · Capstone

The Design Spine: An End-to-End Agent Sketch

Tangible win: a single repeatable template that turns "design an agent" into eight ordered passes — frame, fit, loop, tools, memory, gates, monitoring, failure modes — and you produce one full architecture sketch with it for a real use case, defending every box by the cost it removes.

The One Idea

The mission's hardest success criterion is to "sketch a design with explicit perception, reasoning, action, memory, monitoring, approval, and recovery paths" — and then defend the trade-offs in plain language. Every lesson so far taught one of those surfaces in isolation. This lesson is the spine that holds them together: an ordered checklist you run top to bottom so nothing is forgotten and nothing is over-built.

The discipline is design from the task inward, not from the pattern outward. You do not start with "let's use multi-agent ReAct with a verifier." You start with the goal and add only the machinery the task's cost structure demands. Keep the compact template open while you work: The Design Spine Reference.

Minimum Rule

The smallest reliable design wins. Every pass you add — a second agent, a ReAct loop, a verifier, a gate — buys something specific and costs latency, money, or complexity. You must be able to name what each box removes and defend each box you left out. A capstone you can't defend in plain language isn't finished.

The Eight Passes

Run these in order. Each maps to a lesson you've already done — the spine is mostly retrieval, not new knowledge.

1 · Frame

Goal & boundaries

Goal, environment, percepts (stale? noisy? adversarial?), actions (reversible?), authority, stop condition. 0001, 0002.

2 · Fit

Smallest type + topology

Simplest agent type that meets the goal; single-agent unless work cleanly splits. 0004, 0011.

3 · Loop

Reasoning + planning

One-shot / CoT / ReAct; sequential / parallel / retrieval-first; verifier only where it pays. 0005, 0007, 0012.

4 · Tools

Contracts

Per tool: capability, args, permissions, side effects, cost, failure modes. 0006, 0009.

5 · Memory

Governed store

Episodic / procedural / scratchpad; provenance + expiry; retrieval by authority + recency. 0013.

6 · Gates

Control boundaries

Gate where wrong-action cost > wait cost; pick interrupt mode; checkable triggers; route feedback. 0014.

7 · Watch

Monitoring + recovery

Catch loops/drift/no-progress; recover locally then escalate; rollback path; keep provenance. 0008, 0010.

8 · Fail

Failure modes

Top risks + mitigations, each mapped to a box you drew: stale state, injection, loops, poisoned memory, hidden authority, irreversible side effects.

Worked Example: An On-Call Triage Agent

To see the spine run end to end, here is a compact sketch for one use case — an agent that triages production incidents for a SaaS service. Read it as a model for the shape of your own answer, not as the only correct design.

1 FRAME    Goal: stabilize a firing incident and hand a clean summary to on-call.
          Percepts: alerts, logs, metrics — DELAYED + NOISY; deploy history (authoritative).
          Actions: read logs (rev), restart service (rev), scale up (rev),
                   roll back deploy (HARD to undo), page human (rev), post status (rev-ish).
          Authority: may restart/scale freely; may NOT roll back a deploy alone.
          Stop: incident acknowledged-resolved, or 15 min of no progress.

2 FIT     Goal-based, single agent. No clean skill split that pays for a second agent.

3 LOOP    ReAct — each log/metric observation changes the next action.
          Planning: sequential (steps depend on prior results), not parallel.
          Verify: cross-check "deploy X caused it" against the deploy timeline before acting.

4 TOOLS   get_logs(service, window) -> read-only, cheap.
          restart(service) -> side effect, reversible, idempotent.
          rollback(deploy_id) -> side effect, HARD to undo, needs approval.
          page(team) -> side effect, reversible.
          Each returns structured errors (access-fail vs. empty result).

5 MEMORY  Episodic: past incidents + what fixed them (provenance, recency-ranked).
          Procedural: runbooks. Scratchpad: this incident's timeline.
          Retrieval ranks by authority + recency, filtered by service permission.

6 GATES   restart/scale/page -> escalate-on-trigger (autonomous).
          rollback -> approve-before (hard to undo, high blast radius).
          Triggers: low confidence, conflicting signals, novel service, no progress.

7 WATCH   Watch for restart loops (same service N times), budget/time cap,
          metric drift. Recover: retry read, fall back to runbook, then page.
          Provenance: every claim in the summary links to its log/metric source.

8 FAIL    Stale metrics -> act on resolved state  | mitig: timestamp + freshness check.
          Restart loop                            | mitig: loop monitor + cap (pass 7).
          Poisoned incident memory                | mitig: validated writes, confidence.
          Rollback without authority              | mitig: approve-before gate (pass 6).
          Prompt injection via log contents       | mitig: treat logs as data, not orders.

Notice how passes feed each other: the rollback being hard to undo (pass 1) forces the approve-before gate (pass 6); the restart loop risk (pass 8) is closed by the monitor (pass 7). The spine's value is exactly these cross-links.

Interleaved Check

Each question lives in the same incident-triage system but tests a different pass. The skill is recognizing which surface owns each decision.

Recall Check

From memory, before scrolling up: name the eight passes of the design spine in order, and the one rule that decides whether to add machinery at any pass.

1 Frame (goal, percepts, actions, authority, stop) · 2 Fit (smallest agent type + topology) · 3 Loop (reasoning + planning + verify) · 4 Tools (contracts) · 5 Memory (governed store) · 6 Gates (control boundaries) · 7 Watch (monitoring + recovery) · 8 Fail (failure modes + mitigations). Rule: add machinery only when its cost is less than the cost it removes — smallest reliable design wins, and you must defend every box you drew and every one you left out.

Capstone Drill: Sketch Your Own

Pick a real use case — ideally one from your own work or a target interview. Suggested options if you want one handed to you: a refund-processing agent, a recruiting-inbox triage agent, or a code-review assistant that opens PRs. Run all eight passes. Keep each pass to a few lines — the goal is a complete spine, not a long document.

There is no single answer here — bring this sketch back to the agent and have it pressure-test you. Use the checklist below to self-grade first.

Defense Check

Your sketch is defensible when you can answer each of these in plain language:

  • What makes this system agentic rather than just a generative model with tools?
  • What is the smallest agent type that fits — and why not something simpler?
  • What happens when a tool call partially succeeds?
  • Which state is authoritative when percepts and memory disagree?
  • What memory is useful, and what memory would be dangerous to keep?
  • What would make a verifier independent rather than the same model grading itself?
  • What monitor would catch a tool loop or state drift before it does damage?
  • Which rules are enforced programmatically (a gate or prerequisite) vs. by prompt guidance?
  • For every box you added — what cost does it remove? For every pattern you skipped — why was it overkill?
