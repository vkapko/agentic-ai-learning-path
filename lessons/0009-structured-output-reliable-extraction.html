Lesson 0009

Design The Extraction Contract

Tangible win: turn a vague extraction request into a schema, validation rule, and retry policy that reduces syntax errors without pretending semantics are solved.

The One Idea

Structured output is a contract between the model and the controller. The Claude architecture notes frame tool use with JSON schemas as the most reliable way to get schema-compliant output, while warning that strict schemas remove syntax errors, not semantic errors [Claude structured output notes].

Keep the compact version open: Structured Output and Reliable Extraction Reference.

Minimum Rule

Use schemas to constrain shape, validators to check meaning, and nullable fields to avoid fabrication when the source is silent. Retry only when the error is repairable from the available evidence.

Four Design Moves

Shape

Force a contract

Represent the expected result as a tool input schema, then read the extracted data from the model's tool call.

Absence

Allow null

Make fields nullable or optional when the source may not contain them, so the model is not pressured to invent values.

Meaning

Validate semantics

Check totals, dates, enum consistency, source spans, and cross-field relationships outside the model.

Repair

Retry narrowly

On retry, include the source, failed extraction, and exact validation error; stop if the missing fact is absent.

Tool Choice Is Control Flow

The notes distinguish three controller choices: auto lets the model return text instead of a tool call, any requires some tool call, and forced tool selection requires a specific named tool [tool_choice notes]. For extraction, that choice is architectural: are you asking for a helpful response, requiring any valid schema, or requiring one exact extraction pass before later work?

Choose The Extraction Move

Pick the move that preserves evidence and avoids false certainty.

Recall Check

From memory: what are the three layers in a reliable extraction contract?

Schema for shape, validator for meaning, and retry policy for repairable failures.

Design Drill: Contract Summary

You need to extract a vendor contract summary from uploaded agreements. Draft the schema boundaries and validation rules.

Required fields: vendor_name, source_document_id, extracted_terms, source_spans. Nullable fields: effective_date, expiration_date, renewal_notice_days, termination_fee. Enums: renewal_type with fixed_term, auto_renewal, month_to_month, other plus renewal_type_detail. Semantic validators: expiration date after effective date, notice days nonnegative, each extracted term has a source span, fee currency present when amount exists. Retry when: date is in the wrong field, enum detail is missing for other, source span is malformed, or format normalization failed. Stop when: the document lacks the fact, access failed, or the source text is contradictory and needs human review.

Design Drill: Retry Message

Write the controller feedback for a retry after validation fails because expiration_date is before effective_date.

Validation error: expiration_date must be after effective_date.
Failed extraction: effective_date=2026-04-01, expiration_date=2025-04-01.
Instruction for retry: reread only the term and date sections, keep null for absent dates, and return source spans for both dates.
Stop condition: if the document contains contradictory dates, set conflict_detected=true and explain the conflict.

Architecture Check

  • Which fields are truly required for downstream execution?
  • Which fields should be nullable because the source may omit them?
  • Which categories need an enum plus an other_detail escape hatch?
  • Which validators check meaning beyond schema shape?
  • Which validation failures are repairable with the same source?
  • Which failures require null, partial result disclosure, or human review?
