Lesson 0004

Decide When More Agents Are Worth It

Tangible win: choose between single-agent, centralized multi-agent, decentralized, and hybrid designs, then name the new failure modes introduced by coordination.

The One Idea

Multi-agent architecture is not an upgrade by default. It is a trade: you buy specialization, parallelism, context isolation, local autonomy, or resilience, and you pay with communication overhead, inconsistent state, conflicting goals, harder debugging, and harder auditability. The local architecture notes define a multi-agent system as two or more agents that coordinate, collaborate, compete, or divide work, which means the design now needs protocols for delegation, shared state, conflict resolution, and monitoring [local architecture notes].

Keep the compact version open: System Architecture Choices Reference.

The Minimum Test

Add more agents only when one of these pressures is real: the work needs specialized roles, can run safely in parallel, benefits from isolated context, requires local autonomy, or must continue when one worker fails. If none of those are true, start with one agent and make the control loop observable.

Four Shapes

Single agent

One decision owner

Best for narrow workflows and one clear owner. Watch for the prompt or controller absorbing too many unrelated responsibilities.

Centralized

Coordinator routes

A central controller assigns tasks and enforces policy. Easier to audit, but it can become a bottleneck or single point of failure.

Decentralized

Agents decide locally

Agents plan and act locally. Useful for distributed environments, but governance and conflict resolution become harder.

Hybrid

Central plus local

Central coordination owns goals, policy, and escalation; local agents handle bounded action. The local notes call this common in real-world agentic systems.

Coordinator-Subagent Pattern

Claude's architecture reference describes a hub-and-spoke coordinator-subagent pattern: the coordinator manages inter-subagent communication, error handling, information routing, task decomposition, delegation, result aggregation, and which subagents to invoke [Claude architecture reference]. A key constraint is context: subagents do not automatically inherit the coordinator's conversation history, so the coordinator must pass the facts, source metadata, quality criteria, and task boundaries they need [context passing].

That makes the coordinator the place to defend coverage, provenance, arbitration, approvals, and recovery. Subagents should be narrow, tool-scoped, and structured in their outputs.

Choose the Shape

Pick the smallest architecture that fits the described pressure.

Recall Check

From memory: name two things multi-agent designs can improve, and two things they make harder.

Improves: specialization and parallelism; also context isolation, local autonomy, or resilience when those are real pressures. Makes harder: shared state, communication, conflict resolution, debugging, auditing, provenance, and error recovery.

Design Drill: Research Coordinator

You are designing a multi-agent research system for "AI impact on creative industries." The coordinator assigns only these subtasks: digital art, graphic design, and photography. Every subagent succeeds, but the final report misses music, writing, and film.

Root cause: coordinator decomposition was too narrow. The subagents executed correctly inside bad task boundaries. Fix: require the coordinator to first map coverage dimensions, assign non-overlapping scopes, and ask synthesis to report coverage gaps before final output. Monitor: log original topic, decomposition, assigned scopes, returned findings, omitted areas, and any re-delegation.

Design Drill: Error Propagation

A web-search subagent times out. Which response should it return to help the coordinator recover?

Return fields: failure type, attempted query, source type, retry count, partial results, suspected cause, alternative query or source, and whether the result is incomplete or empty. Coordinator response: retry with a narrower query, route to another source, proceed with a coverage note, or escalate. Do not mark a timeout as a successful empty result.

Architecture Check

  • What pressure justifies more than one agent?
  • Which decisions are centralized, delegated, local, or human-approved?
  • What exact context must each subagent receive?
  • What structured output must each subagent return?
  • How do agents report partial failure without hiding gaps?
  • Where are shared state, provenance, arbitration, and final stop conditions owned?
