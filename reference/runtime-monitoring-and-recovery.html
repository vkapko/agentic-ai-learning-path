Reference 0008

Runtime Monitoring and Recovery

Use this sheet to decide what to observe during an agent run, when to intervene, and how to recover without hiding uncertainty.

The Decision

A runtime safety net is the controller-owned monitoring and recovery design for an agent run. It records what happened, checks whether the run is still allowed and useful, and chooses a bounded recovery path when execution degrades.

Five Signals

  • Trace: model calls, tool calls, validated inputs, outputs, approvals, retries, and final state.
  • Outcome: success criteria, partial results, blocked reasons, and unresolved assumptions.
  • Risk: policy conflict, low confidence, unsafe action, stale state, sensitive data, or approval threshold.
  • Resource: latency, token use, tool cost, retry count, queue time, and budget exhaustion.
  • Recovery: retry, replan, fallback, escalate, answer partially, rollback, or stop.

Error Categories

Transient

Try again

Timeout, rate limit, temporary service error, or unavailable dependency. Use bounded retries.

Validation

Fix input

Missing ID, malformed argument, invalid enum, or schema mismatch. Repair inputs or ask for data.

Permission

No authority

Caller lacks access or approval. Route to the right authority rather than retrying blindly.

Business

Valid denial

Policy says no, order is ineligible, or no matching record exists. Explain or escalate if policy is ambiguous.

Recovery Rules

  • Retry only when the error is retryable, and cap the retry count.
  • Do not convert failed reads into empty successful results.
  • Escalate when policy, authority, safety, customer request, or progress limits require it.
  • Disclose missing evidence when answering from partial results.
  • Turn repeated near-misses into design work: tool contracts, routing, prompts, tests, or policy gates.

Minimum Log Event

  • Identity: task ID, user/session boundary, agent version, model version, and correlation ID.
  • Decision: selected action, reason code, policy result, confidence or verifier result.
  • Tool: tool name, validated input summary, output summary, latency, cost, and error category.
  • Recovery: retry count, fallback chosen, escalation target, partial evidence, and final disclosure.

Review Question

Before shipping an agent workflow, ask: if this run fails halfway through, will the controller know what happened, what is still safe to do, and what must be disclosed to the user?
