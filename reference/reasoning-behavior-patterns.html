Reasoning Behavior Patterns

Use this sheet to choose the smallest reasoning pattern that fits the workflow pressure.

The Decision

Do not add a reasoning pattern because it sounds more agentic. Add it because the workflow has a specific pressure: multi-step decomposition, live observation, pre-release review, failure recovery, or reusable feedback from prior attempts.

Pattern Picker

One-shot

Bounded generation

Use for low-risk answers or artifacts where no external state is needed. Validate output before delivery.

Decomposition

Multi-step thinking

Use when breaking the task into steps improves reliability. Prefer hidden reasoning or concise summaries over raw reasoning traces.

ReAct

Reason, act, observe

Use when tool results or environment observations affect the next step. The controller owns tool execution and loop termination.

Reflection

Critique and revise

Use when a draft, decision, or failed action needs a review pass. It is a quality aid, not a correctness proof.

Episode memory

Reuse feedback

Use when human or environment feedback should shape later attempts. Store scope, source, date, confidence, and expiry.

Controller Responsibilities

  • Expose only the tools allowed for the current role and task.
  • Validate tool names, arguments, permissions, and side effects before execution.
  • Represent tool errors as structured observations, not silent empty results.
  • Track task state separately from long-term memory.
  • Log model calls, tool calls, observations, approvals, revisions, and final outcomes.
  • Stop on success, escalation, refusal, max turns, max cost, unsafe action, or unrecoverable error.

Warning

A visible reasoning trace is not evidence that the answer is correct. For auditability, preserve structured inputs, outputs, tool calls, observations, verifier results, and source provenance.
