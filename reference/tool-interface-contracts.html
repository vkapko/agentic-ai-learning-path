Reference 0005

Tool Interface Contracts

Use this sheet to make agent tools selectable, permissioned, validated, observable, and recoverable.

The Decision

A tool contract is the boundary between model intent and system effect. The model may request a tool, but the controller validates the request, enforces policy, executes the action, trims the result, logs the event, and decides whether the observation returns to model context.

Contract Template

  • Name: short verb-noun name that matches the real capability.
  • Purpose: when to use it, when not to use it, and how it differs from similar tools.
  • Input schema: required fields, optional fields, nullable fields, enums, formats, and validation rules.
  • Output schema: task-relevant result fields, confidence or provenance fields, and omitted raw data location if needed.
  • Authority: allowed callers, required task state, permission checks, approval gates, and rate limits.
  • Side effects: read-only, reversible, externally visible, expensive, irreversible, or safety-sensitive.
  • Failure schema: category, retryability, attempted input, partial result, user-safe message, and recovery hint.
  • Telemetry: caller, validated arguments, result summary, latency, cost, retries, errors, approvals, and correlation ID.

Error Categories

Validation

Bad request

Argument is missing, malformed, contradictory, or outside schema. Usually ask for correction or route to clarification.

Permission

Not authorized

Caller lacks access, approval, scope, or task state. Escalate or request approval; do not retry blindly.

Business

Rule blocked

The request is valid but violates domain policy, such as a refund window or account status rule.

Transient

Try later

Timeout, rate limit, or temporary dependency issue. Retry with limits and preserve partial results.

Anti-Patterns

  • Returning an empty successful result for a timeout or access failure.
  • Letting the model enforce financial, privacy, or safety approvals by prompt only.
  • Exposing broad write tools when a read-only or narrow tool would fit.
  • Using similar tool names and descriptions without examples or boundaries.
  • Sending verbose raw tool output into context when a narrow result shape is enough.

Review Question

If this tool is misused, what is the worst credible outcome: wrong answer, wasted cost, privacy leak, external action, financial loss, safety issue, or silent coverage gap? Design the contract around that risk.
