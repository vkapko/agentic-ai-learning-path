Reference 0014

Human-in-the-Loop and Approval Boundaries

Use this sheet to place human gates where they earn their cost, choose the interrupt mode, write checkable escalation criteria, and route human feedback to the system that should own it.

The Rule

Gate an action when the cost of a wrong autonomous action exceeds the cost of waiting for a human — and nowhere else. Every gate names a trigger, an interrupt mode, and an accountable owner. Gating everything dies of latency; gating nothing is unsafe.

Three Interrupt Modes

Approve-before

Block, then act

Pause until a human approves. For irreversible, unsafe, or expensive actions. Pays full latency every time.

Review-after

Act, then correct

Act now, human reviews later, corrections feed back. For reversible, high-volume, latency-sensitive work.

Escalate-on-trigger

Run until a line is crossed

Autonomous until a defined trigger fires, then hand off. The default for everything in between.

Teaching / demonstration is a fourth flavor: a human example is captured as a procedural memory or skill template — a record, not a weight change.

Escalation Triggers

Standard, checkable conditions that should route a decision to a human:

  • Low confidence — model or tool confidence below a set threshold.
  • Safety risk — the action could cause harm.
  • Missing permission — the caller lacks authority for the action.
  • Irreversible action — a wrong move can't be undone.
  • Policy conflict — the action collides with a rule or constraint.
  • Novel environment state — a situation the agent has not handled.

What Feedback Changes (Three Updates)

  • Memory write — a scoped record (e.g. a client preference). Cheap, instant, reversible.
  • Policy / threshold update — a rule binding every run (e.g. an approval cap). Reviewed and rolled out once.
  • Model-weight update — RLHF or fine-tuning. Slow, expensive; not what one correction does.

A demonstration does not make the model permanently learn a skill by itself. It may produce a procedural memory, a skill template, a threshold/prompt/routing change, or a separate component fix. See Memory reference for the same frame.

Anti-Patterns

  • Gating everything "to be safe" — latency and staffing cost make the agent useless.
  • Review-after on an irreversible action — the review can't undo it.
  • Vague triggers ("if it seems risky") instead of checkable booleans.
  • Feedback leaking into prompts without validation — one stray correction becomes durable behavior.
  • No audit trail — no one can say who approved or overrode a gated action.
  • Treating one demonstration as if the model permanently learned the skill.
  • No fallback when the human is slow or absent, so the agent stalls or guesses.

Review Question

For each gate: does wrong-action cost beat wait cost, is the interrupt mode matched to reversibility and volume, and when feedback arrives, is it routed to memory, policy, or training — with who-approved-what logged?
