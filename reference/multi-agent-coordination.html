Reference 0011

Multi-Agent Coordination

Use this sheet to pick a coordination shape, scope worker roles and tools, and design the shared state that keeps every agent reading one truth.

The Decision

A coordination pattern answers three questions: who decomposes the goal, who does the work, and where the single source of truth lives. Pick the lightest shape that matches the task's dependency structure.

Three Shapes

Fan out

Orchestrator-worker

One orchestrator decomposes, delegates to narrow-tool workers, and merges results. For parallelizable subtasks with centralized tracking. Risks: orchestrator bottleneck, inconsistent results, distributed retries.

Deliberate

Expert team

Domain agents collaborate on one decision behind a single interface. For tasks needing multiple forms of domain reasoning. Needs disagreement, source-ranking, and final-authority protocols. Risks: cost, latency, groupthink.

Nest

Hierarchical

Orchestrators over sub-orchestrators over workers and teams. For very large or nested workflows. Scales coordination, not reliability by itself. Risks: overhead, state drift, slow diagnosis.

Selection Test

  • Flat fan-out of independent subtasks → orchestrator-worker.
  • One decision needing several domains of reasoning → expert team.
  • Too large for one orchestrator to track → hierarchical.
  • None of these pressures → stay single-agent (see lesson 0004).
  • Reliability comes from observability, retries, idempotency, and failure isolation — not from adding layers.

Worker Contract

  • Role: one narrow responsibility per worker.
  • Tools: least-privilege tool set for that role.
  • Returns: structured result, error packet, or status — never free-text only.
  • Merge: orchestrator validates schema, attaches sources, rejects unparseable returns.
  • Failure: bounded retry, then coverage gap and partial proceed or escalation.

Collective Memory Record

Shared-State Rules

  • Treat memory as data infrastructure, not a chat log.
  • Every record carries provenance, owner, timestamp, confidence, and expiry.
  • Enforce access control, retention, and explicit conflict resolution.
  • Resolve conflicts by ownership and authority, not recency alone.
  • A central store can be a source of truth — and a bottleneck or single point of failure.
  • Guard against prompt injection through stored content.

Review Question

Before adding agents or layers, ask: does the dependency structure require this shape, and where does the one source of truth live?
