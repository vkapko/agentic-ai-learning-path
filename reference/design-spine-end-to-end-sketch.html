Reference 0015

The Design Spine

A one-page, repeatable template for sketching an agentic system. Walk the eight passes in order; each names the design decisions and the prior reference that owns the detail. Stop at the smallest reliable design — every added pass is a cost, not a virtue.

The Rule

Design from the task inward, not from the pattern outward. Pick the smallest agent type, the simplest reasoning loop, and the fewest gates that make wrong actions survivable. You should be able to defend every box by the cost it removes — and defend every box you left out.

1 · Frame the task

  • Goal — the outcome, not the steps.
  • Environment & percepts — what the agent observes, and which inputs are stale, delayed, noisy, partial, or adversarial.
  • Actions — what can change the environment; flag each reversible or not.
  • Authority — what the agent may do without asking.
  • Stop condition — what ends the run (success, budget, no progress).

Detail: Agentic Control Loop.

2 · Pick the smallest fit

  • Agent type — reflex → model-based → goal-based → utility-based. Choose the simplest that meets the goal.
  • Topology — single agent first; multi-agent only when work splits by skill or context cleanly and the coordination cost pays for itself.

Detail: System Architecture Choices, Multi-Agent Coordination.

3 · Choose the reasoning + planning loop

  • Reasoning — one-shot, chain-of-thought, or ReAct. Use ReAct only when each observation changes the next action.
  • Planning — sequential, parallel (independent steps), or retrieval-first (ReWOO) when info can be gathered up front.
  • Verification — add an independent second-pass check only where a wrong answer is costly and external evidence exists.

Detail: Reasoning Patterns, Tool Orchestration, Search, Verification & Self-Correction.

4 · Specify the tools

For each tool, write the contract: capability, arguments, permissions, side effects, cost, latency, and failure modes (with structured, retryable errors). Two similar tools need descriptions distinct enough to route correctly.

Detail: Tool Interface Contracts, Structured Output.

5 · Design the memory

  • What's stored — episodic (events), procedural (skills/runbooks), scratchpad (this run). Not the raw transcript.
  • Record metadata — provenance, owner, timestamp, confidence, expiry.
  • Retrieval — rank by authority + recency (+ permission filter), not similarity alone.

Detail: Memory Is Data Infrastructure.

6 · Place the control boundaries

  • Gates — gate an action only when wrong-action cost > wait cost.
  • Interrupt mode — approve-before (irreversible/expensive), review-after (reversible/high-volume), escalate-on-trigger (the default).
  • Triggers — checkable booleans: low confidence, safety risk, missing permission, irreversible action, policy conflict, novel state.
  • Feedback routing — memory write vs. policy/threshold update vs. model-weight update; log who approved or overrode.

Detail: Human-in-the-Loop & Approval Boundaries.

7 · Add monitoring + recovery

  • Watch — tool loops, state drift, budget/latency, unsafe escalation, no progress.
  • Recover locally first — retry, fall back, then escalate; distinguish access failure from a valid empty result.
  • Rollback / compensation — what undoes a partial success.
  • Provenance — preserve claim-source mappings through synthesis.

Detail: Runtime Monitoring & Recovery, Context, Escalation & Provenance.

8 · List the failure modes

Name the top risks and their mitigation: stale state, prompt injection, tool loops, poisoned memory, weak verification, hidden authority, irreversible side effects, slow/absent human. Each should map to a box you already drew (or a gap you must close).

The Defense Test

You can defend the sketch when you can answer, in plain language: what makes it agentic rather than generative; the smallest agent type that fits; what a partial tool success does; which state is authoritative; what memory is useful vs. risky; what would make the verifier independent; what catches a tool loop; and which rules are enforced programmatically vs. by prompt.
