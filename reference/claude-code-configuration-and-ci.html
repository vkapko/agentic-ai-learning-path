Reference 0016

Claude Code Configuration and CI/CD

Use this sheet to place each Claude Code instruction at the right surface and sharing level, choose plan mode vs direct execution, and wire a trustworthy CI review.

The Rule

Place each instruction at the narrowest scope that still covers everywhere it must apply, and at the sharing level that matches who must follow it. Two questions decide everything: when does it load, and who gets it.

The Five Surfaces

Root CLAUDE.md

Always loaded, shared

Universal project standards. Permanent context cost — keep to what applies everywhere.

Directory CLAUDE.md

Loaded in subtree, shared

Conventions for one area/package. Use @import to pull shared standards and stay modular.

.claude/rules/

On matching paths, shared

YAML paths globs load the rule only when editing matching files — even across many directories.

.claude/skills/

On demand

Task workflow. context: fork isolates output; allowed-tools scopes authority; argument-hint prompts inputs.

.claude/commands/ — slash commands, on demand, project (shared) or user (personal). Hierarchy: user (~/.claude/, never shared via repo) → project (root or .claude/) → directory.

When to Use Which

  • Applies everywhere, always → root CLAUDE.md.
  • Applies to one folder → directory CLAUDE.md (or a rule if it may spread).
  • Applies to a file type scattered across the tree → path-scoped rule with a glob.
  • On-demand workflow → skill (verbose/destructive → add context: fork / allowed-tools).
  • Saved prompt → command (project = shared, user = personal).
  • Team must follow it → project scope (committed). Only you → user scope.

Use /memory to see which files are actually loaded and diagnose inconsistent behavior across sessions.

Plan Mode vs. Direct Execution

  • Plan mode — large-scale, multi-file, architectural, or multiple-valid-approach tasks; explore and design before changing, to avoid costly rework.
  • Direct execution — well-scoped changes with a clear path (single-file fix, known stack trace).
  • Combine — plan to investigate, then execute the plan. Use the Explore subagent to isolate verbose discovery and return a summary.

CI/CD Integration

  • -p / --print — non-interactive; processes prompt, writes stdout, exits. Without it the pipeline hangs.
  • --output-format json + --json-schema — machine-parseable structured findings for automated PR comments.
  • CLAUDE.md — supplies review criteria, testing standards, fixtures to the CI-invoked agent; cuts low-value output.
  • Independent instance — review runs fresh, not as the session that wrote the code. A generator reviewing itself is a weak verifier. Re-runs: include prior findings, report only new or unaddressed issues.

Anti-Patterns

  • Team standard placed at user scope (~/.claude/) — teammates never receive it.
  • Always-loaded CLAUDE.md bloated with conventions that apply only to some files — a path-scoped rule would load them only when relevant.
  • Copies of a directory CLAUDE.md to cover files scattered across the tree — use one glob rule instead.
  • Plan mode on a trivial single-file fix (overhead) — or direct execution on a 45-file migration (rework).
  • CI review reusing the generating session — not an independent verifier.
  • Running CI without -p — the pipeline hangs waiting for input.

Review Question

For each instruction: is it at the narrowest scope that still covers everywhere it applies, and at the sharing level matching who must follow it? Is plan mode reserved for big/architectural work? Does the CI review run with -p, structured output, CLAUDE.md criteria, and as an independent instance?
