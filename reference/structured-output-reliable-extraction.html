Reference 0009

Structured Output and Reliable Extraction

Use this sheet to design extraction contracts that separate output shape, semantic validation, and retry policy.

The Decision

A structured extraction contract defines what data shape the model must produce, what missing information may be represented as null, what semantic validators run outside the model, and which failures deserve a retry.

Schema Rules

  • Required: fields needed for downstream execution, audit, or routing.
  • Nullable: fields that may be absent from the source but still matter downstream.
  • Enums: closed categories used for routing, reporting, or policy decisions.
  • Other detail: escape hatch for real categories outside the enum.
  • Source spans: evidence locations for extracted claims that need auditability.

Validator Rules

Shape

Schema valid

Types, required fields, enum values, and nested object structure are accepted.

Meaning

Semantics valid

Totals match, dates are ordered, currencies are present, and field relationships make sense.

Evidence

Source valid

Important values have source spans or excerpts, and absent values are marked null rather than guessed.

Retry Rules

  • Retry format, placement, enum-detail, normalization, or malformed source-span errors.
  • Include the source document, failed extraction, and exact validation error.
  • Do not retry indefinitely; cap attempts and record the final failure reason.
  • Do not retry when the source lacks the requested fact; return null or escalate if the field is essential.
  • Escalate contradictory source evidence, access failures, policy-sensitive decisions, or repeated semantic mismatch.

Minimum Extraction Contract

{
  "schema": "required, nullable, enum, other_detail, source_span",
  "validator": "cross-field meaning checks outside the model",
  "retryPolicy": "retry only repairable errors with exact feedback",
  "disclosure": "return null, conflicts, partial coverage, and review needs"
}

Review Question

Before shipping an extraction workflow, ask: if the JSON is perfectly valid but the values are wrong, what catches the problem before downstream systems trust it?
