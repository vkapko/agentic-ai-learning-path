Reference 0006

Tool Orchestration Plans

Use this sheet to decide which tool calls run now, which wait, which require approval, and how results are synthesized.

The Decision

A tool orchestration plan is the controller-owned execution map for a multi-tool workflow. It marks known inputs, dependencies, parallel groups, gates, failure paths, and the final evidence bundle returned to the model.

Plan Template

  • Goal: user-visible outcome and stop condition.
  • Known inputs: identifiers, permissions, task state, constraints, and missing facts.
  • Parallel group: independent read or retrieval calls that can run together.
  • Dependencies: calls that need outputs from earlier calls.
  • Gates: policy, verifier, confidence, budget, safety, or human approval checks.
  • Failure paths: retry limits, partial-result behavior, fallback, escalation, and audit notes.
  • Synthesis: final response shape, evidence list, omitted raw data, and correlation ID.

Execution Rules

Parallel

Independent reads

Run together when calls have all required inputs, do not depend on each other, and do not mutate shared state.

Serial

Data dependency

Run after when a call needs an ID, fact, permission, or decision produced by a prior step.

Gate

Sensitive effect

Require explicit policy or human approval for irreversible, unsafe, expensive, externally visible, or low-confidence actions.

Synthesize

Evidence bundle

Send the model a compact, provenance-rich summary rather than every raw tool payload.

Anti-Patterns

  • Parallelizing calls that compete for the same mutable record.
  • Letting a prompt reminder act as the only approval gate for a sensitive tool.
  • Returning a final answer as if a failed parallel branch had succeeded.
  • Using a general DAG compiler for a two-step workflow where fixed sequencing is clearer.
  • Passing bulky raw tool outputs into context instead of a narrow evidence bundle.

Review Question

Before adding another model call, ask: is this actually a reasoning problem, or is it an execution-shape problem the controller can solve with dependencies, gates, retries, and synthesis?
