System Architecture Choices

Use this sheet to decide when a single agent is enough, when coordination is worth it, and which decisions must stay centralized.

The Decision

Start with one agent unless the workflow has a real coordination pressure: specialized roles, parallelizable work, local autonomy, independent actors, or context isolation. The local architecture notes warn that multi-agent systems require communication, delegation, shared state, conflict resolution, and monitoring.

Architecture Options

Single agent

One owner

Use for narrow workflows with one clear decision owner. Risk: one prompt or controller accumulates too many responsibilities.

Centralized

Coordinator controls

Use when governance, observability, or high-level routing must be consistent. Risk: bottleneck, central failure, and heavy coordination traffic.

Decentralized

Agents decide locally

Use when local responsiveness, fault isolation, or intermittent connectivity matters. Risk: conflicting decisions and harder governance.

Hybrid

Central plus local

Use when central policy owns goals and escalation, while local agents handle bounded decisions. This is common in real deployments.

Coordinator-Subagent Rule

A coordinator should own task decomposition, delegation, result aggregation, error routing, and final arbitration. Subagents should receive explicit context, role-specific tools, and a structured output contract. Do not assume subagents inherit the coordinator's conversation, memory, or source context automatically.

Failure Modes Added by More Agents

  • Coverage gap: the coordinator decomposes the task too narrowly, so every subagent succeeds but the result misses part of the problem.
  • State split: agents act from different versions of facts, policy, or task status.
  • Duplicate work: agents investigate the same area because scopes are unclear.
  • Conflict: two agents recommend incompatible actions or optimize different goals.
  • Hidden failure: a timeout or access failure is reported as an empty result instead of structured error context.
  • Lost provenance: synthesis drops source URLs, document names, page numbers, dates, or confidence notes.

Sketch Template

Coordinator owns: goal, decomposition, routing, shared state, arbitration, approval, final answer. Subagents own: bounded task, allowed tools, local retries, structured findings. Shared contract: input context, output schema, source metadata, error schema, stop condition.
